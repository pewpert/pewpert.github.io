A full-stack rental aggregator built for my own move to Tokyo — and a deep lesson in why production scraping is an engineering discipline, not a weekend task.

The What

A web application that aggregates Tokyo rental listings from three major portals — Suumo, LIFULL HOME'S, and Chintai — based on user-specified criteria: ward, rent range, room type, floor size, and walk time to station.

Users submit a search, the backend queues a Celery job, and a headless Chromium browser (via Playwright) scrapes the portals in parallel. Results pass through a filter pipeline, are stored in PostgreSQL, and displayed in a sortable table in a Next.js frontend. A demo mode with five hardcoded seed listings is live on Vercel — the full stack runs locally via Docker Compose.

The Why

I'm relocating to Tokyo in 2026. Japan's rental market is notoriously difficult to navigate from abroad: listings are spread across competing portals, UI is Japanese-only, and search filters don't translate cleanly between sites. The same apartment can appear on three platforms with slightly different prices and availability.

Generic apartment search tools don't understand Tokyo-specific constraints — earthquake building standards (post-1981 new code), foreigner acceptance policies, the gap between advertised rent and true move-in cost once deposit, key money, and agency fees are added. I needed something that could enforce my exact criteria across all three portals in one pass.

Building it myself also meant I'd understand the data. By the time the scraper ran, I knew the Suginami/Nakano Marunouchi Line corridor was the correct answer — not because an algorithm told me, but because I'd read enough listings to see the pattern.

How It Was Built

The scraping layer was the core challenge. Each portal has a different HTML structure, pagination pattern, and anti-scraping posture. LIFULL HOME'S and Chintai work with Playwright and realistic browser headers. Suumo was harder — repeated test runs from a Docker container during development triggered a temporary IP block. The code was correct; the problem was behavioural. You can't brute-force scraper development against a live site without consequences.

That experience pointed toward Firecrawl — a managed scraping API that handles browser rendering, proxy rotation, and anti-bot measures as a service. Rather than maintaining headless Chrome configuration and fighting IP blocks, Firecrawl exposes a clean scrape(url) function that returns structured markdown or HTML. The tradeoff is cost per request versus the engineering overhead of doing it yourself. Firecrawl is wired into the environment as the next implementation step.

The rest of the stack was more straightforward: FastAPI for the backend API, Celery + Redis for async job queuing (so the frontend doesn't block waiting for scrapes that can take 30–60 seconds), and PostgreSQL for result storage. The full local environment runs as five Docker Compose services. The Next.js frontend was rebuilt with a compact filter bar, real-time debug panel showing per-source raw vs. filtered counts, and a sortable listings table.

What I Learned

  • Scraping is a maintenance contract, not a one-time build. Portal HTML structures change without notice. Suumo's Chintai URL parameter format, Homes' ward codes, Chintai's table layout — all of these required reverse-engineering from scratch. The working scrapers today are one site update away from returning empty results. Structural health checks that alert instead of silently failing are not optional.

  • Managed scraping APIs change the cost equation. Playwright is powerful but operationally heavy — browser process management, proxy configuration, IP reputation, and anti-bot cat-and-mouse are all your problem. Firecrawl's scrape() function trades per-request cost for all of that complexity. For a personal tool scraping a few dozen pages a day, the managed approach is almost always the right call.

  • Async job queues change the UX problem. A scrape that takes 45 seconds is fine — if the user isn't staring at a loading spinner. Celery + Redis let the frontend submit, show a "queued" state, and poll for results. The engineering overhead of adding a job queue is real, but the alternative (synchronous scraping behind a single API call) fails in production the moment a portal is slow.

  • Filter pipeline quality matters as much as data collection. Chintai returned 112 raw listings for my base criteria — and zero after filtering. The portal accepts rent parameters in the URL but doesn't enforce them strictly; all filtering has to happen post-scrape. A raw data pipeline that hands back unfiltered results isn't useful; the value is in the enforcement layer, not the fetch layer.
